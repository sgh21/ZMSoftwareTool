从启发式自动分类 → 配置文件驱动的显式配色映射

1.3 当前问题：端盖识别不全

UR10 的 7 个 DAE 文件共包含 20 个子网格，但 CAP_GEOMETRY_NAMES 仅硬编码了 4 个：

base.dae （1 子网格）

base.dae ← 端盖未识别

shoulder.dae （3 子网格）

shoulder.dae shoulder.dae_2 ✓ shoulder.dae_3

upperarm.dae （4 子网格）

upperarm.dae upperarm.dae_2 upperarm.dae_3 upperarm.dae_4 ← 端盖未识别

forearm.dae （5 子网格）

wrist1.dae （3 子网格）

wrist1.dae wrist1.dae_2 ✓ wrist1.dae_3

wrist2.dae （3 子网格）

wrist2.dae wrist2.dae_2 ✓ wrist2.dae_3

wrist3.dae （1 子网格）

wrist3.dae ✓

对于 base、upperarm、forearm 这三个部件的端盖子网格，因为不在 CAP_GEOMETRY_NAMES 中，且其 DAE 材质色不满足 "blueish" 阈值（B>R×1.18），导致它们被错误分类为 long_link 或 joint 色，显示为白色或灰色，而非蓝色。

二、重构方案：配置文件驱动的显式配色映射

2.1 核心思路

放弃启发式自动推断，改为在 theme.yaml 中显式声明每个子网格的颜色类别。程序启动时读取配置，直接查表获得颜色，无需任何形状分析或颜色阈值判断。

旧方案：启发式推断

def _resolve_mesh_color(mesh, geometry_name):
    # 1. 检查硬编码集合
    if geometry_name in CAP_GEOMETRY_NAMES:
        return "cap"
    # 2. 读 DAE 材质色
    # 3. 计算伸长率
    # 4. 判断 blueish
    # 5. 判断亮度
    # ... 20 行 if/elif

问题 依赖 DAE 材质色和形状比例，不可靠；新增模型需改代码

新方案：配置表驱动

def _resolve_mesh_color(mesh, geometry_name):
    category = mesh_color_map.get(geometry_name)
    if category:
        return self._render_colors[category]
    return self._render_colors["silver"]

优势 精确、可配置、可扩展、不依赖 DAE 材质色

2.2 配置文件设计

在 config/theme.yaml 中新增 mesh_color_map 节点：

# config/theme.yaml
status_colors:
  alarm: "#db3444"
  normal: "#45db34"
  warning: "#db7734"

robot_render_colors:
  long_link: "#F0F2EB"
  joint:     "#6E7570"
  connector: "#6E7570"
  cap:       "#9ED6F5"
  silver:    "#E0E3DB"

# 新增：子网格 → 颜色类别 的显式映射
mesh_color_map:
  # === base (底座) ===
  base.dae:       long_link      # 注意：base 整个 mesh 包含端盖和底座身体，
                                 # 如果 DAE 无法拆分，则需整体归为一类
  # === shoulder (肩部) ===
  shoulder.dae:   long_link      # 肩部主体连杆
  shoulder.dae_2: cap            # 肩部蓝色端盖 ✓
  shoulder.dae_3: joint          # 关节连接件

  # === upperarm (上臂) ===
  upperarm.dae:   long_link      # 上臂主体
  upperarm.dae_2: cap            # 上端盖（当前未识别为蓝色）
  upperarm.dae_3: cap            # 下端盖（当前未识别为蓝色）
  upperarm.dae_4: joint          # 关节连接件

  # === forearm (前臂) ===
  forearm.dae:    long_link      # 前臂主体
  forearm.dae_2:  joint          # 关节连接件
  forearm.dae_3:  cap            # 上端盖（当前未识别为蓝色）
  forearm.dae_4:  cap            # 下端盖（当前未识别为蓝色）
  forearm.dae_5:  connector      # 连接件

  # === wrist1 ===
  wrist1.dae:     long_link      # 腕部1主体
  wrist1.dae_2:   cap            # 蓝色端盖 ✓
  wrist1.dae_3:   joint          # 关节连接件

  # === wrist2 ===
  wrist2.dae:     long_link      # 腕部2主体
  wrist2.dae_2:   cap            # 蓝色端盖 ✓
  wrist2.dae_3:   joint          # 关节连接件

  # === wrist3 ===
  wrist3.dae:     cap            # 末端工具法兰（蓝色端盖）✓

⚠ 上表中的 upperarm.dae_2/3、forearm.dae_3/4 等具体映射为示例值，实际哪个子网格是端盖需要检查 DAE 文件中的 geometry name 与 visual shape 的对应关系来确认。

2.3 如何确认精确的子网格映射

方法一：日志调试法

在 _iter_visual_meshes() 中临时加一行 print(geometry_name, color)，运行程序后观察控制台输出，列出所有 20 个 geometry_name 及其当前颜色。根据输出调整 YAML 中的映射。

方法二：调试着色模式

给每个子网格分配独一无二的调试色（如用 hash 生成），在渲染画面中逐个对比确认各子网格对应的物理位置。

方法三：Blender 直接查看 DAE

用 Blender 导入每个 DAE 文件，在 Outliner 面板中查看子网格名称（shape0, shape1, shape2...），对照 yourdfpy 的命名规则（filename.dae, filename.dae_2, filename.dae_3...），视觉上确认端盖对应哪个 shape。

3.2 核心代码变更

robot_simulation_widget.py 变更对比：

# ===== 删除以下内容 =====
CAP_GEOMETRY_NAMES = {
    "shoulder.dae_2",
    "wrist1.dae_2",
    "wrist2.dae_2",
    "wrist3.dae",
}

# ===== _resolve_mesh_color 重写前 (25行) → 重写后 (8行) =====
def _resolve_mesh_color(self, mesh, geometry_name: str = "") -> tuple[float, float, float]:
    category = self._mesh_color_map.get(geometry_name)
    if category and category in self._render_colors:
        return self._render_colors[category]
    return self._render_colors["silver"]

# ===== 新增方法 =====
def _load_mesh_color_map(self) -> dict[str, str]:
    """从 theme.yaml 加载 mesh_color_map 配置"""
    theme_path = self.project_root / "config" / "theme.yaml"
    if not theme_path.exists():
        return {}
    try:
        data = yaml.safe_load(theme_path.read_text(encoding="utf-8")) or {}
    except Exception:
        return {}
    mapping = data.get("mesh_color_map", {}) if isinstance(data, dict) else {}
    if not isinstance(mapping, dict):
        return {}
    return {str(k): str(v) for k, v in mapping.items()}

四、实施步骤

  1. 确认子网格映射关系

    用日志调试法运行程序，收集所有 20 个 geometry_name，对照 Blender 或渲染画面确认每个子网格对应哪个物理部件（连杆主体 / 端盖 / 关节连接件）。这是最关键的一步，映射错了就会配错色。

  2. 编写 theme.yaml 配置

    在 config/theme.yaml 中添加 robot_render_colors（目前只有默认值）和完整的 mesh_color_map。每个 geometry_name 都要有对应条目。

  3. 修改 robot_simulation_widget.py

    删除 CAP_GEOMETRY_NAMES 常量，删除 _resolve_mesh_color 中的启发式逻辑，新增 _load_mesh_color_map() 方法，在 __init__ 中调用加载。

  4. 更新测试

    运行 pytest tests/，检查 test_default_project_loads_ur10_and_debug_angles 中的颜色断言是否仍然通过。根据新的映射调整期望值。

  5. 视觉验证

    启动应用，加载 UR10 模型，旋转视角检查所有端盖是否都显示为蓝色。重点检查 base、upperarm、forearm 的端盖区域。

五、方案优势总结

对比维度旧方案（启发式）新方案（配置驱动）
准确性 不可靠 依赖 DAE 材质色和形状 精确 显式声明，所见即所得
可维护性 差 修改映射需改代码 好 改 YAML 即可，无需重新编译
可扩展性 受限 新机器人型号需调启发式参数 灵活 新模型只需新增一段 YAML
支持多模型 不支持 启发式耦合 UR10 特征 支持 可按 URDF 路径配置多套映射
代码复杂度 ~25 行判断逻辑 ~8 行查表逻辑
配置复杂度 0 配置（看似简单） ~20 行 YAML 映射

可选增强：按 URDF 路径切换配色方案

如果未来需要支持多种机器人型号，可以在 theme.yaml 中按 URDF 文件名组织多套映射：

mesh_color_map:
  ur10:
    base.dae:       long_link
    shoulder.dae_2: cap
    # ...
  ur5:
    base.dae:       long_link
    # ...

加载时根据当前 URDF 文件名选择对应的映射表。当前阶段仅支持 UR10，暂不需要此复杂度。
